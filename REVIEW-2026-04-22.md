## Bottom line

**[fact]** The code works as a v0.5 prototype. Encryption is real, playbooks load safely, the basic loop is there.
**[read]** It is not close to being a tool a real person should run against their real PII today without understanding what they're getting into.
**[read]** The gap between "DeleteMe replacement" and what this actually is right now is **large**. The README oversells.

Two separate review lenses:
- **Sonnet's audit** caught implementation bugs: format-string injection, duplicate listings, confirmation-link phishing, silent-failure CAPTCHAs, broken test fixtures.
- **DeepSeek's red-team** caught the **framing problem**: this is not a crypto library, it's a weapon in an adversarial ecosystem with legal, ethical, and collective-action dimensions the audit completely ignored.

Both are right. Both matter.

## The top 10 action items (merged, ranked)

### Must fix before anyone else runs this (security/correctness)

1. **Format-string injection in `extra_fields` and `email_body_template`** (`brokers/web_form.py`, `brokers/email.py`)
   - A malicious community-PR'd playbook can exfiltrate the whole Profile object via `{first.__class__.__init__.__globals__}` traversal.
   - Fix: replace `str.format()` with `string.Template.safe_substitute()` on all playbook-provided templates.
   - This has to land before you ever accept a third-party playbook PR.

2. **Confirmation-link phishing in `inbox/imap.py` + `runner.py:check_inbox`**
   - Current regex matches any `https?://...` in email bodies. A phishing email with "Opt Out Confirmation" in the subject gets its link clicked automatically.
   - Fix: check link domain matches the playbook's `homepage` domain before `page.goto()`.

3. **URL scheme + domain validation in `playbook.py`**
   - `url_template` and `opt_out.url` accept `file://`, `javascript:`, and arbitrary exfil domains.
   - Fix: Pydantic validator requiring `https://` and the URL's registered domain matching `homepage`'s registered domain.

4. **Test fixtures don't exist**
   - `tests/test_playbook_validation.py` references `FIXTURES / "valid_playbook.yaml"` and `"invalid_playbook.yaml"` which aren't in the repo. CI is broken on a fresh checkout.
   - Fix: create the fixtures, make CI actually pass.

5. **Duplicate listing insertion in `state.py`**
   - No UNIQUE constraint on `(broker, listing_url)`. Every rescan multiplies the rows and files duplicate opt-outs.
   - Fix: add UNIQUE + INSERT OR IGNORE.

### Must fix before public launch (integrity/credibility)

6. **Silent-failure detection for scans** (`brokers/web_form.py`)
   - Right now: selector rotates or Cloudflare blocks = "0 listings found" = user thinks they're clean.
   - Fix: log a warning when all selectors match 0 on a successfully loaded page, and when the HTML is suspiciously short (likely a block page).

7. **CAPTCHA detection with a real headed-mode pause**
   - BUILD_SPEC claims this is handled. It isn't. Cloudflare/hCaptcha/reCAPTCHA break the big 3 brokers silently.
   - Fix: detect "challenge"/"captcha" markers, in headed mode call `input("solve it and press enter")`, in headless emit a clear error.

8. **SECURITY.md has two lies**
   - Says "AES-256" (Fernet is AES-128).
   - Says `state.db` contains no PII (listing URLs have name/city/state embedded).
   - Fix: rewrite with literal truth. Privacy tools cannot afford to be factually wrong about their own crypto.

9. **File permissions on `profile.yaml.enc`, `state.db`, `network.log`**
   - All written with default umask. On Linux they're world-readable by default.
   - Fix: `os.chmod(path, 0o600)` after every write.

10. **LEGAL.md that honestly addresses CFAA/ToS/SLAPP**
    - The repo has no legal disclosure at all right now. First HN lawyer comment will savage it.
    - Sonnet's take: CFAA risk is low post-*Van Buren*, disclose that plainly.
    - DeepSeek's correction: that's glib. The real legal exposure is SLAPPs against maintainers, not individual users. Disclose both.

## The framing problems DeepSeek surfaced (harder, more important)

These aren't line-number bugs. These are "what is this project actually?" questions.

### A. The threat model is backwards

Sonnet worried about malicious playbooks stealing user PII. DeepSeek worried about **PrivacyWorm being used as a weapon against a third party**:
- **Stalker use case**: wipe a victim's contact info from people-search sites so concerned family can't find them.
- **False-flag opt-outs**: submit opt-outs for a celebrity/enemy using their PII (most brokers don't verify requester identity strongly).
- **DoS on broker support**: no rate limits means a bad actor can weaponize the tool for mass fake submissions.

The tool has no identity tie-in between "who runs the tool" and "whose profile is encrypted on disk." Possession of the profile is 100% of the law.

**[read] This is the single most serious gap.** The honest answer might be: add a signed attestation during `init` that says "I am the person this profile represents, or I am their authorized agent." Not legally binding, but makes the malicious-use case premeditated in a way that matters if you ever testify in a stalking case.

### B. Adversarial arms race, not a bug list

Brokers don't have static websites you need to parse correctly once. They're adaptive antagonists whose business model opposes easy automated opt-outs.

- **Honeypots**: Hidden form fields and fake "submitted successfully" pages that silently flag the request as bot-originated and drop it. Playbooks can't detect this.
- **Behavioral fingerprinting**: Playwright's default Chromium has a detectable fingerprint. Brokers can identify + permanently blacklist the tool's pattern across all users.
- **Data laundering**: Broker A deletes the user → Broker B, who bought the data yesterday, re-sells it to Broker A's next scrape. The hydra problem.

**[read] What this means for the roadmap:** any v1 success metric based on "form submitted" is meaningless theater. You need "record verified removed on next scan" as the real success signal. And you need to accept that the long-run equilibrium is: brokers win unless legislation forces them.

### C. Global legal reality, not just US

Sonnet's review was CCPA + CFAA heavy. Ignored:
- **GDPR Article 17** (right to erasure) is stronger than any US law. If this tool targets EU brokers it needs playbooks that format requests as formal Art. 17 requests.
- **LGPD, PIPEDA, PIPA** etc. in other markets.
- **SLAPPs**: CFAA vagueness is the perfect weapon for a broker to sue a high-profile user or the maintainer into bankruptcy. PR fallout is irrelevant when the goal is attrition through legal costs.

### D. Local-only is a feature AND a weakness

Local-only buys you no telemetry, no lock-in, no central honeypot. It also costs you:
- **Playbook updates**: users need `git pull` skills, or you ship an auto-updater that breaks "local-only."
- **No collective intelligence**: 1000 users can't tell you "Spokeo's selector broke today" automatically.
- **State divergence**: two machines = two uncoordinated state databases = duplicate or missed requests.

**[read] An honest v2 design might split:** the tool stays local-only for PII, but ships an optional anonymous telemetry channel that just reports "selector X on broker Y matched 0 elements" so playbook maintainers get a heartbeat. That's a hard trust-model call, but "no telemetry ever" is also a choice with real operational costs.

### E. The sustainability question

DeleteMe employs humans because maintaining 750 playbooks for rapidly changing adversarial websites is genuinely a full-time job. An OSS community may not scale to match it, which means the project is perpetually "almost working" and the README's implied parity with DeleteMe is aspirational at best, misleading at worst.

**[read]** Either (a) reframe PrivacyWorm as a 20-broker power-user tool and drop all "DeleteMe alternative" language, or (b) build infrastructure for community playbook testing/freshness monitoring/selector auto-heal, which is a multi-year investment.

## What to do with this

Two paths:

**Path 1 — ship it small and honest.**
- Fix items 1-5 above.
- Rewrite README: "automates opt-out form submission for 10 brokers, on your own computer, with your own eyes on every step." Drop DeleteMe comparisons.
- Ship a 60-line `LEGAL.md`.
- Post to HN as "a weekend project that files opt-outs for the 10 biggest people-search sites." Nobody expects DeleteMe parity from that framing. You ship, you learn, no legal target painted on the project.

**Path 2 — build the real thing.**
- Fix items 1-10 above AND design for DeepSeek's systemic risks.
- Address the weaponization case (identity attestation, rate limiting by design, abuse-reporting mechanism).
- Build telemetry-free selector freshness signals (e.g., a separate "canary runner" repo that maintainers run, not users).
- Accept that this is a 2-year project and a potential legal target.
- Possibly incorporate as a nonprofit before publishing (liability shield + legitimacy).

**[guess]** Path 1 is the right call given Simon's bandwidth and focus-layer commitments. This is a satisfying side-project but it's not a Groundlayer business. If you ship it honestly small, it's a credible portfolio piece + community contribution. If you ship it as "the open-source DeleteMe" it's a legal risk + a credibility hit when selectors rot.
